## Review Swift Code

`docs/coding_rule.md` のコーディング規約に基づいて Swift プロダクトコードをレビューする。

### 使い方

引数としてファイルパスまたはディレクトリパスを受け取る。
引数がない場合は `git diff --name-only` で変更された `.swift` ファイルを対象にする。
`BefoldApp/befoldTests/` 配下のファイルはスキップする（テストコードは `review-swift-tests` スキルで扱う）。

### Steps

1. `docs/coding_rule.md` を読み込み、レビュー基準を把握する。
2. 対象ファイルを特定する:
   - 引数にファイルパスがあればそれを使う
   - 引数にディレクトリがあれば配下の `.swift` ファイルを列挙する
   - 引数がなければ `git diff --name-only HEAD` で変更ファイルを取得する
   - `befoldTests/` 配下は除外する
3. 各ファイルを読み込み、以下のチェックリストに沿ってレビューする。
4. 発見事項をファイルごとに報告する。

### チェックリスト

#### Concurrency モデル

- [ ] UI 状態を持つクラスに `@MainActor` が付いているか
- [ ] データバインディングに `@Observable` を使っているか（`ObservableObject` + `@Published` は不可）
- [ ] GCD キューでスレッド安全性を保証するクラスに `@unchecked Sendable` が付いているか
- [ ] スレッド境界を越えるクロージャに `@Sendable` が付いているか
- [ ] `nonisolated(unsafe)` がプロダクトコードで使われていないか（テストコードのみ許容）

#### 型設計

- [ ] 継承を意図しないクラスに `final` が付いているか
- [ ] 外部読み取り可能・書き込み不可のプロパティに `private(set)` を使っているか
- [ ] インスタンス化不要な分類型は `enum` で定義しているか
- [ ] 値型の `enum` に `Sendable` 準拠があるか
- [ ] `switch` 文で `default` を使わず全ケースを明示しているか

#### 命名規約

- [ ] 型名が UpperCamelCase になっているか
- [ ] メソッド・プロパティが lowerCamelCase になっているか
- [ ] GCD キューラベルがリバースドメイン形式か（`com.degino.befold.*`）

#### パターン

- [ ] オプショナルのアンラップに `guard let ... else { return }` を使っているか
- [ ] クロージャで `[weak self]` + `guard let self else { return }` を使い循環参照を防いでいるか
- [ ] ファイル内セクションに `// MARK: - <セクション名>` を使っているか

#### コメント・ドキュメンテーション

- [ ] 公開クラス・公開メソッドに `///` ドキュメンテーションコメント（日本語）があるか
- [ ] 非自明な判断・ワークアラウンドにインラインコメント（日本語）があるか
- [ ] コードをそのまま訳しただけの不要なコメントがないか
- [ ] タスク番号や変更履歴への参照がコメントに残っていないか

#### AppKit / SwiftUI 混在ルール

- [ ] ウィンドウ管理は AppKit（`NSWindowController`）で行っているか
- [ ] ビューコンテンツは SwiftUI（`View` プロトコル）で定義しているか
- [ ] WKWebView ラッパーが `NSViewRepresentable` + `Coordinator` パターンを使っているか

#### WKWebView / JavaScript ブリッジ

- [ ] JS に渡す文字列を `JSONEncoder` でエスケープしているか（XSS 防止）
- [ ] 不要な再描画を防ぐ差分チェックがあるか
- [ ] JS 関数名・メッセージハンドラ名・注入スクリプトが `ViewerBridge` に集約されているか
      （`evaluateJavaScript` への文字列リテラル直書きは違反）

#### 依存注入・定数の単一情報源

- [ ] 外部依存（ファイル読込・ネットワーク・ファイル監視・タイマー）は既存プロトコル
      （`FileReading` / `ReleaseFetching` / `UpdateDownloading` / `FileWatching`）と同じ方針で
      イニシャライザ注入されているか。新しい外部依存を具象直参照で増やしていないか
- [ ] 注入はデフォルト引数付きで、既存呼び出し元を変更せずに済む形か
- [ ] メソッド内部で依存を直接 `new` していないか（注入済みプロパティ/ファクトリを使う）
- [ ] 対応拡張子は `FileType.allExtensions` を参照しているか（拡張子リストの再定義は違反）
- [ ] 正規化パスキー・rename 移行は `URL.normalizedPathKey` / `PathKeyedDictionary` を
      経由しているか（キー生成規約の再実装は違反）

#### エラーハンドリング

- [ ] `guard` + early return で異常系を先に処理し、正常系のネストを浅く保っているか
- [ ] ファイル読み取り失敗を適切にフォールバックしているか

#### コード品質（SwiftLint 閾値）

- [ ] 循環的複雑度が 10 以下か
- [ ] 関数の行数が 50 行以下か
- [ ] ファイルの行数が 400 行以下か
- [ ] クラス/構造体の行数が 250 行以下か
- [ ] 行長が 120 文字以下か

### 出力フォーマット

ファイルごとに発見事項を以下の形式で報告する:

```
## <ファイルパス>

### 問題 (問題がある場合)

- **<カテゴリ>**: <具体的な指摘> (L<行番号>)

### 良い点 (特筆すべきものがある場合)

- <規約に沿った良い実装の具体例>
```

最後に全体のサマリーを付ける:

```
## サマリー

- レビュー対象: <N> ファイル
- 問題: <N> 件（重大: <N>, 軽微: <N>）
- 規約準拠度: <高/中/低>
```
