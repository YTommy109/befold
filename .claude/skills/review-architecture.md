## Review Architecture

Swift コードベースのアーキテクチャ品質をレビューする。
「共通化・抽象化の観点でレビューして」という依頼もこのスキルで扱う
(チェックリストの「共通化・抽象化」「DI・テスタビリティ」に重点を置く)。

### 使い方

引数なしで実行する。コードベース全体を対象にする。

### Steps

1. dagayn でリードを収集する:
   - `architecture_analysis_tool(mode="overview")` — ハブノード・SAP 違反・knowledge_gaps
   - `refactor_tool(mode="suggest")` — split(責務分割)候補と行数/分岐数メトリクス
2. 実地調査をサブエージェントへ **単一メッセージで並列に** 委譲する:
   - dagayn 出力が大きい場合はその解析専任を 1 体
   - プロダクトコードの読解(重複・抽象化不足・依存注入)を 1 体
   - テストコードと同梱リソース(viewer.html / viewer.js / CSS)を 1 体
3. グラフのリードと実地調査の結果を突き合わせる。両方が一致した指摘を優先する。
4. チェックリストに沿って観点ごとに評価し、報告する。

### dagayn 出力の扱い

- **remove(デッドコード)候補は原則偽陽性として裏取りする。**
  NSApplicationDelegate / NSWindowDelegate のコールバック、NSViewRepresentable 要件
  (makeNSView / updateNSView / makeCoordinator / dismantleNSView)、@objc セレクタ、
  Swift Testing の @Test 関数は、フレームワークが動的に呼ぶため静的グラフに
  caller が現れない(caller_count=0 は削除根拠にならない)。
  フレームワーク契約を確認できたものだけを報告し、それ以外は報告に含めない。
- split 候補の行数・分岐数は信頼できる生データとして使う。
- SAP 違反(D > 0.5)と stable_concrete_pressure は「安定パッケージに抽象が無い」
  リード — DI・テスタビリティ観点の具体確認先として使う。
- answerability が degraded(missing_flows / missing_communities)の場合、
  グラフ由来の主張には graph-limited と明記する。

### チェックリスト

#### レイヤー設計

- [ ] パッケージ間の依存方向が上位→下位の一方向か
- [ ] 循環依存が存在しないか
- [ ] 各パッケージの責務が明確に分離されているか
- [ ] 安定依存の原則(SDP)に違反していないか — 不安定なパッケージに安定なパッケージが依存していないか

#### 関心の分離

- [ ] 各クラス/構造体が単一責任原則を守っているか
- [ ] ファイル行数が過度に大きくないか(目安: 200行超で要確認)
- [ ] 複数の関心が1つのクラスに混在していないか(例: UI + ビジネスロジック、ライフサイクル + メニュー構築)
- [ ] ヘルパー/ユーティリティの抽出が適切か

#### 共通化・抽象化

- [ ] 同型のコードが 2 箇所以上にないか(永続化の骨格、HTTP 検証、確認ダイアログ、
      Process 起動、rename 移行等)。重複は **両方の箇所を file:line 付きの対で引用** する
- [ ] 同じイベントへの反応が複数層に手続きコピーされていないか(例: rename の旧→新差し替え)
- [ ] 同一の定数・マジック文字列が複数ファイルに散在していないか
      (UserDefaults キー、拡張子集合、JS 関数名・メッセージ名、URL、遅延秒数)
- [ ] 言語をまたぐ定数の二重定義がないか(例: Swift の ZoomStore と viewer.js の ZOOM_MIN/MAX)。
      ある場合、ソースを読んで整合を検証するテストが存在するか
- [ ] 共通化の提案が偽の抽象になっていないか — データ形状・目的が異なるものを
      無理に統合しない。見送る場合は理由を「見送り判断」に記録する

#### DI・テスタビリティ

- [ ] 外部依存(ファイルシステム、ネットワーク、タイマー等)がプロトコルで抽象化されているか
- [ ] 具象型の直接生成がテスタビリティを阻害していないか
- [ ] イニシャライザインジェクションまたはファクトリパターンが使われているか
- [ ] プロダクション呼び出し元がデフォルト引数で既存コードを変更せずに済むか
- [ ] 既存の抽象(プロトコル注入)と同じ方針で拡張できるか — 新規の抽象は既存パターンに揃える

#### データフロー

- [ ] データの伝搬経路が明確か(コールバック / @Observable / Combine 等)
- [ ] スレッド安全性が確保されているか(@MainActor, GCD キュー, ロック等)
- [ ] 不要な再処理を防ぐ差分チェック/デバウンスがあるか

#### テスト責務の分離

- [ ] Unit Test が外部依存(ファイルシステム、ネットワーク等)に依存していないか
- [ ] Integration Test と Unit Test が明確に分離されているか(ファイル単位 or Suite 単位)
- [ ] E2E テストの方針が明確か(自動化 or 手動チェック)
- [ ] テスト未対象のビジネスロジックが残っていないか
- [ ] モック/スタブが適切に使われているか

### 証拠規律

- 指摘には必ず file:line と短いコード引用を付ける。
- グラフ由来のリードは実ソースで裏取りしてから報告する(裏取りできないものは
  「graph-limited なリード」と明記して区別する)。
- テスト未カバーの主張は `query_graph_tool(pattern="tests_for")` か実テストの確認を経てから書く。

### 出力フォーマット

観点ごとに発見事項を以下の形式で報告する:

```
## <観点名>

### 評価: ✅ 良好 / ⚠️ 懸念 / ❌ 問題

<具体的な指摘と根拠(file:line を含む)>

### 改善提案(懸念/問題の場合)

- <具体的な改善案>
```

最後に以下の 3 点を必ず付ける:

```
## サマリー

| 観点 | 評価 | 要約 |
|---|---|---|
| レイヤー設計 | ✅/⚠️/❌ | <一行要約> |
| 関心の分離 | ✅/⚠️/❌ | <一行要約> |
| 共通化・抽象化 | ✅/⚠️/❌ | <一行要約> |
| DI・テスタビリティ | ✅/⚠️/❌ | <一行要約> |
| データフロー | ✅/⚠️/❌ | <一行要約> |
| テスト責務の分離 | ✅/⚠️/❌ | <一行要約> |

## 優先度つき改善提案

効果 × リスクで順位付けし、各項目に一行の根拠を添える(1. 〜 N.)。

## 見送り判断

共通化・抽象化を意図的に見送った項目とその理由
(偽の抽象になる / 検証が弱まる / スコープ外)。無ければ「なし」と書く。
```
