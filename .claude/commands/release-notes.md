# /release-notes — リリースノート生成

前回タグからのコミットを元にリリースノートを生成する。

引数: $ARGUMENTS（dev | stable）— 省略時は対象タグ（最新タグ）に `-` が
含まれるかどうかで自動判定する（含む: dev / 含まない: stable）。

## 1. タグの取得

```bash
git tag --sort=-v:refname | head -2
```

- 最新タグと前回タグを取得する
- タグが 1 つしかない場合は初回リリースとして全コミットを対象にする

## 2. コミットログの取得

```bash
git log {前回タグ}..{最新タグ} --pretty=format:"%s" --no-merges
```

## 3. リリースノートの生成

生成方針は dev / stable で異なる。

### dev（pre-release）の場合

これまで通り、Conventional Commits のプレフィックスで全コミットをグループ化して
Markdown 形式で出力する:

```markdown
## {最新タグ}

### Features
- 項目（feat: から）

### Bug Fixes
- 項目（fix: から）

### Others
- 項目（上記以外）
```

該当がないグループは省略する。

### stable（一般リリース）の場合

**ユーザーに影響のある内容だけ** を対象にする。開発・運用に関する内容（CI/CD、
テスト、リファクタリング、ドキュメント、ビルド設定、依存関係の更新、内部ツール
など）は、コミットの種類に関わらず含めない。

1. 各コミットについて、ユーザー（アプリの利用者）から見て挙動や体験が変わるか
   を判断する。
   - 含める例: 新機能の追加、UI/UX の変更、ユーザーが遭遇する不具合の修正、
     パフォーマンス改善（体感できるもの）
   - 除外する例: `chore:` `test:` `docs:` `ci:` `refactor:` `style:` の接頭辞、
     および `feat:`/`fix:` であっても内部実装・開発体験・CI・テストのみに
     関する変更（例: 「quality-loop のイテレーション上限を明確化する」
     「コーディング規約を更新する」）
2. 判断に迷うコミットはユーザー影響がないものとして除外する（安全側に倒す）。
3. 除外したコミットがある場合は、リリースノート生成後にユーザーへ一覧を
   提示し、含めるべきか確認する。
4. 残ったコミットを Markdown 形式で出力する:

```markdown
## {最新タグ}

### 新機能
- 項目

### 修正
- 項目
```

該当がないグループは省略する。ユーザー影響のあるコミットが 1 件もない場合は
その旨をユーザーに報告し、リリースノートの内容について確認を取る。
