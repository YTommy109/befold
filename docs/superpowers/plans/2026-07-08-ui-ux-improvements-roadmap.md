# UI/UX 改善ロードマップ

> このドキュメントは12件の要望を横断的に俯瞰するための**ロードマップ**であり、
> writing-plans スキルが定めるビルド可能な TDD 実装計画そのものではない。
> 各グループ／項目は独立したサブシステムに属するため、着手時に
> `docs/superpowers/specs/YYYY-MM-DD-<name>-design.md` と
> `docs/superpowers/plans/YYYY-MM-DD-<name>.md` を個別に作成し、
> superpowers:writing-plans の通常フォーマット（ヘッダー・タスク分解・TDD手順）で
> 実装計画に落とし込むこと。

## グルーピングと優先度

要望を影響範囲・実装レイヤーで4グループに分類した。グループ内は独立性が高いため
並行着手も可能。グループ間も依存はないが、ユーザー体験への影響度が高い順に
G1 → G4 を推奨順とする。

### G1: ウィンドウ/ファイル操作の基本動作（優先度: 高）

| # | 要望 | 現状の関連コード |
|---|------|----------------|
| 1 | ファイルを開いた時にアクティブウィンドウにならない | `BefoldApp/befold/App/ViewerWindowManager.swift`（ウィンドウ生成・管理）, `AppDelegate.swift` |
| 2 | cmd+o のディレクトリをウィンドウ単位で記憶、未オープン時はホーム | `AppDelegate.swift:127-136`（`showOpenPanel`）, `BefoldApp/befold/Viewer/DirectoryLister.swift:118-123`（`resolveFileToOpen`） |
| 6 | パスコピーをフルパスでなく相対パスに | `BefoldApp/befold/Viewer/FileListView.swift:177-187`（`copyFileReference` / `copyPath`） |

**論点（着手前に要確認）:**
- #1: `NSWindow` 生成後に `makeKeyAndOrderFront` 相当の呼び出しが漏れている、もしくは
  ウィンドウ生成が非アクティブな `NSApp` コンテキストで行われている可能性が高い。まず
  `ViewerWindowManager` のウィンドウ生成箇所と `NSApp.activate` の呼び出し有無を確認する。
- #2: 「ウィンドウ単位」の記憶場所をどこに持たせるか（`ViewerWindowManager` が
  ウィンドウごとの状態を持っているなら拡張、なければ新規プロパティが必要）。
  「ウィンドウが開いていない場合はホームディレクトリ」という要件も明文化する。
- #6: 「最初に開いたファイルのディレクトリをルート」の定義を確定する
  （ウィンドウ単位のルートディレクトリ = #2 の記憶先と共有できる可能性がある）。

### G2: キーボード/ジェスチャー操作（優先度: 高）

| # | 要望 | 現状の関連コード |
|---|------|----------------|
| 3 | スペースキーでスクロール、shift+スペースで逆スクロール | `BefoldApp/befold/Resources/`（viewer.html / viewer.js）, `ViewerWebView.swift` |
| 4 | 上下矢印・vim jk でスクロール、モディファイヤーで大きくスクロール | 同上 |
| 10 | ズームジェスチャー対応 | `BefoldApp/befold/App/ZoomStore.swift`, 既存 `docs/superpowers/plans/2026-07-04-diagram-zoom.md`（ズーム機能は実装済み、ジェスチャー入力の追加） |
| 11 | 二本指ジェスチャーで履歴を遡る | 該当する履歴/ナビゲーション機構は未確認 → 要調査（ファイル間の前後移動機能が存在するか） |

**論点:**
- #3/#4 はどちらも WKWebView 内のキー入力処理に関わるため、同じ実装単位
  （viewer.js のキーハンドラ拡張）にまとめられる可能性が高い。1つの計画にまとめて良い。
- #10 は既存のズーム基盤（`ZoomStore`）にマウス/トラックパッドの `NSEvent.magnification`
  ジェスチャーを配線する作業。ズームの UI（パネル等）は diagram-zoom 計画で既に整理済みなので
  そのコンテキストを参照すること。
- #11「履歴を遡る」機能が現状存在するか要確認。存在しない場合はまず「ファイル間の
  戻る/進む」履歴機能自体の設計が必要になり、G2の中でも最も大きいタスクになる可能性がある。

### G3: レンダリング品質（優先度: 中）

| # | 要望 | 現状の関連コード |
|---|------|----------------|
| 5 | base64 エンコード画像がソースコードモードで表示される不具合 | `BefoldApp/befold/Resources/viewer.js`, 関連仕様 `docs/superpowers/specs/2026-07-05-svg-source-toggle-design.md`（レンダリングソース切替機能） |

**論点:**
- ソース表示トグル機能（SVG source toggle）実装時に、base64 データURIを含む画像が
  誤って「ソースコード」として判定されている可能性が高い。まず
  `docs/superpowers/plans/2026-07-05-svg-source-toggle.md` の実装内容を読み、
  判定ロジック（拡張子/内容判定）のどこで base64 埋め込みを見落としているか特定する。
  これは既存機能のバグ修正であり、新規設計ドキュメントは不要でそのまま
  systematic-debugging → 修正計画で対応可能。

### G4: ブランディング/デザイン（優先度: 中〜低、まとめて対応可）

| # | 要望 | 現状の関連コード |
|---|------|----------------|
| 7 | GitHub Pages に著作権表記（Tommy109） | `docs/index.html:180-184`（`<footer>`、現在著作権表記なし） |
| 8 | About の Degino → Tommy109、リンクを GitHub プロフィールに | `BefoldApp/befold/App/AppDelegate.swift:150-164`（`aboutPanelOptions`、`"Degino Inc."` → `https://www.degino.com/`） |
| 9 | サイドドロワーの発見性向上（細い縦バー＋ホバーで自動展開） | `BefoldApp/befold/Viewer/FileListView.swift`、既存 `docs/superpowers/plans/2026-07-05-file-list-sidebar.md`（トグル機構は実装済み） |
| 12 | サイドバーのデザイン刷新（角丸・配色） | 同上 `FileListView.swift` |

**論点:**
- #7/#8 は文言・リンク差し替えのみで実装コストは低い。#8 は GitHub プロフィール URL
  （`https://github.com/YTommy109` などユーザーが使う実際のプロフィール）を確認してから着手する。
- #9/#10 の詳細挙動（バーの幅、ホバー判定領域、自動展開のアニメーション、角丸の具体的な値・
  新配色）は brainstorming スキルでデザイン仕様を先に固めてから実装計画を書くべき
  （「デザインが古い」という主観的フィードバックは具体的な色/角丸値に落とし込む必要がある）。

## 次のステップ

1. G1・G3 は仕様がほぼ自明なので、次のいずれかから着手する実装計画（TDD形式）を作成する。
2. G2 の #11（履歴機能の有無）と G4 の #9/#12（デザイン仕様の具体化）は、実装計画を書く前に
   brainstorming スキルで要件を詰める必要がある。
3. どのグループ／項目から着手するか、ユーザーに確認して1件ずつ
   `docs/superpowers/specs/*.md` + `docs/superpowers/plans/*.md` を作成し、
   subagent-driven-development または executing-plans で実行する。

### `/quality-loop` で見つかった未対応の指摘（次回対応）

<!-- blocked-by #未着手次回セッションへ持ち越し -->

- **[bug] 二本指スワイプが横スクロール可能なコンテンツと衝突しうる**
  (`BefoldApp/befold/App/ViewerWindowController.swift` の `handleScrollWheelForHistorySwipe`)
  幅広の mermaid 図・横長コードブロック・テーブルなど、WKWebView 内で本来横スクロール
  したいコンテンツに対しても、横優勢のスワイプであれば無条件で履歴ナビゲーション
  （戻る/進む）に化けてしまう。縦優勢ガードは縦スクロールとの衝突しか防げず、
  横スクロールしたい場面ではむしろ誤発火しやすい。
  対応方針の候補: (1) イベント対象のスクロールコンテナが実際にその方向へスクロール
  余地を持つかを判定に加える、(2) WebKit 標準の `allowsBackForwardNavigationGestures`
  を活かしつつ別の仕組み（例: メニュー/キーボードショートカット）でファイル履歴に
  フックする設計に見直す。ユーザー判断により今回は対応を見送り、次回検討する。
